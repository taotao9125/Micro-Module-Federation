介绍 #

微模块 #

从 2016 年微前端概念提出至今，业界已具有相对成熟的现成技术方案。从概念上来讲，微前端是解决前端应用组织协作的方案，从技术上来讲，指在宿主应用里面加载子应用。各应用由相应的团队独立开发、独立部署，解决团队之前协作耦合。

市面上已有相对成熟的微前端技术方案，如基于 single-spa 的 qiankun，基于 WebComponent 的 micro-app，它们以子应用 html 为为起点，以技术手段加载子应用以及依赖的资源，并做到各应用之间相互隔离，互不影响。

我们不妨从后端微服务概念去思考，微服务有一个很重要的特性：每个微服务都可以是一个服务生产者，也可以是一个服务消费者。服务生产者需要给服务消费者提供特定的接口去消费。

对应到前端，以模块的角度去看待“子应用”，远程 host 作为模块生产者，宿主 host 作为模块消费者，远程只负责提供模块，这个模块从组织上可大到一个应用，小到一个函数。远程生产者 host 只负责提供模块，至于怎么消费，那是宿主 host 的职责。这种模式可称为「微模块」。

什么是 Micro Module Federation #

Micro Module Federation 是基于 webpack5 module federation（全文简称 MF) 特性的微模块解决方案，在 webpack5 内置 MF 的功能基础上做了渐进式增强，如错误处理以及模块隔离特性，旨在不同应用之间可自由共享模块。

下面是是一个基本的示例：

js
  <zmf-app
    beforeMount = {() => {console.log('beforemount')}}
    onMount = {() => {console.log('onMount')}}
    onUnmount = {() => {}}
    onError = {() => {}}
    name="yourAppName"
    module="remoteAppContainerName/RichText"
  />

Micro Module Federation 也采用了 WebComponent 方案，如果远程共享模块在业务定义上是一个 UI 组件，就会被挂载限制在 shadowdom 里，同时隔离远程组件的样式、 window、document 访问或操作，避免远程与宿主之间相互冲突影响。

Micro Module Federation 的优势 #

  1. 可加载远程应用里的任意模块。无论这个模块从组织意义上是一个组件还是一个子应用。
  2. 提供了 js 、style、dom 操作隔离等特性，并且这是可选的，并非强制。

Micro Module Federation 的劣势 #

  1. 依赖 webpack5。
